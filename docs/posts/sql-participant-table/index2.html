Research question

Even before you start analyzing your data and think about model or even start grant application, we need to first check whether we have enough data to do that.

In this post, suppose that we want to test feasibilty of our thought (association between sleepd disorders and incident Alzhdimer’s disease risk). The goal of this post is to test feasibilty: how many patients meet basic cohort criteria, and what are their simple demographics?

For this example, I used three synthetic EHR tables: patients, diagnoses, and encounters. The data are fully synthetic and are intended to mimic common relational EHR structures. You can create your own using R code that I shared and found in : … (add path or repositories)

The final output will be:

  1. A cohort attrition table
  2. A simple demographic summary of the final feasible cohort.

Cohort definition

First of all, we need to define our cohort. For this example, we will define our cohort as follows:

  • Has at least one sleep disorder diagnosis.
  • Index date is the first sleep disorder diagnosis date.
  • Age is at least 20 years old at index date.
  • Has no Alzheimer’s disease diagnosis before or on the index date.
  • Has at least 365 days of observed follow-up after index date.

This is the minimum criteria for our cohort. In practice, you may have more complex criteria, but the logic will be similar.

Load data into SQLite

Code 
# Libraries for this project, you can ignore as it is mostly for data manipulation and database connection in R environments. -----
library(DBI) # connect to databases and send data or queries to them.
library(RSQLite) # SQLite database driver for R
library(dplyr) # data manipulation and analysis
library(readr) # read CSV files into R

con <- dbConnect(SQLite(), ":memory:") # create an in-memory SQLite database for demonstration purposes. In practice, you would connect to your actual database.

# I save the synthetic data to CSV files for demonstration purposes. In practice, you would load your actual EHR data from your database or other sources.
patients <- read_csv("data/synthetic_patients.csv") %>%
  mutate(BIRTH_DATE = as.character(BIRTH_DATE),
         DEATH_DATE = as.character(DEATH_DATE))
diagnoses <- read_csv("data/synthetic_diagnoses.csv") %>%
  mutate(DX_DATE = as.character(DX_DATE))
encounters <- read_csv("data/synthetic_encounters.csv") %>%
  mutate(ENCOUNTER_DATE = as.character(ENCOUNTER_DATE))

dbWriteTable(con, "patients", patients, overwrite = TRUE)
dbWriteTable(con, "diagnoses", diagnoses, overwrite = TRUE)
dbWriteTable(con, "encounters", encounters, overwrite = TRUE)

This creates a temporary SQLite database in memory and loads the 3 synthetic EHR tables into it.

Data structure

This project uses three synthetic EHR-style tables. The important point is that the tables do not have the same unit of analysis.

Table Unit of analysis Key columns Role in feasibility test
patients One row per patient PATID, BIRTH_DATE, DEATH_DATE, SEX, RACE Defines baseline demographics and death date
diagnoses Multiple rows per patient PATID, ICD10, ICD10_TXT, DX_DATE Defines sleep disorder index date and Alzheimer’s disease diagnosis
encounters Multiple rows per patient PATID, ENCOUNTER_ID, ENCOUNTER_DATE, ENCOUNTER_TYPE Defines EHR activity, baseline lookback, and follow-up

The three tables are linked by PATID.

This structure matters because joining diagnosis or encounter records directly to the patient table can create duplicate patient rows. For feasibility assessment, the final output must return to one row per patient.
